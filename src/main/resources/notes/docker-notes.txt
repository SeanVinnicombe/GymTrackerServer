What is Docker
==================================

Dokcer is a containerization platform that packages up an application with all its dependencies so the application runs
quickly and reliably from one computing environment to another. Therefore, there is no need to manually go to each of
the servers to install and run the application. You already know that the application has everything it needs to run
and therefore removes the tedious process of setting up the environment.

Two main Concepts:

1. Image - the blueprint. This is a read-only package containing your application code, runtime, dependencies and
configuration. Think of it as a Java class-it's the definition, not the running thing.
2.Container - a running instance of the image. Basically an object instantiated from a class. You are able to run
multiple containers from the same image.

Image (blueprint)  →  Container (running instance)
    Class          →      Object
    Recipe         →      Meal

This is important because:
* You build an image once, and then you can run it any number of times.
* You can run it anywhere - your laptop, colleagues' machine or a cloud server.
* You can run 10 containers from the same image simultaneously for scaling.

What is actually happening inside a container?
-----------------------------------------------------------------

A Container contains:
* Your application code and compiled artifacts
* The exact runtime needed (Java 22, for example)
* All library dependencies
* Environment configuration
* Its own filesystem
* Its own network interface

Though most importantly, it shares host machines OS kernel. This is what makes containers lightweight compared to VMs.

What is the OS Kernel?
===============================================

Think of OS having two layers:

┌─────────────────────────────────┐
│         User Space              │
│  (your apps, Chrome, IntelliJ,  │
│   terminal, file explorer)      │
├─────────────────────────────────┤
│         Kernel Space            │
│  (the kernel — core of the OS)  │
├─────────────────────────────────┤
│         Hardware                │
│  (CPU, RAM, disk, network card) │
└─────────────────────────────────┘

The kernel is the core of the OS. It's that software that sits between your applications and the physical hardware.
It's the first thing that loads when your computer starts and last to shut down.

What does it actually do?
----------------------------------------

Everything that your application needs from the hardware has to go through the kernel.The kernel decides what memory
to allocate and where.

* Memory - when your Spring Boot application needs RAM, it asks the kernel. The kernel decides what memory to allocate
and where.

* CPU scheduling - your machine might have 8 cores but 200 processes running. The kernel decides what processes run on
which core and for how long. This is important when it comes to threads as the kernel is what actually switches between
threads

* File system - when you read a file, your code calls something like 'Files.readAllBytes()'. That eventually becomes a
kernel call that physically reads bytes from the disk.

* Network - when app opens a socket connection to PostgreSQL, the kernel manages the actual networkcoommunication

* Device management - keyboard, mouse, disk, network card - all managed by kernel via drivers.

Application never talks to hardware directly, everything goes through kernel:

                        Your Java code
                              ↓
                             JVM
                              ↓
                        Operating System APIs
                              ↓
                            Kernel
                              ↓
                        Physical Hardware


Why kernel is important for Docker
-------------------------------------------

Windows, Linus and macOS all have different kernels. Though Linus containers specifically rely on features built into
the Linux kernel called:

* namespaces - what gives each container its own isolated view of the system
(its own filesystem, network and its own process list)

* cgroups(control groups) - limits the resources(CPU/memory) that a container can use.

When Docker runs a container, it's not running a separate OS. It's using these Linux kernel features to create an
isolated slice of the existing kernel.

                            Container A         Container B
                            ────────────        ────────────
                            App A               App B
                            Its own filesystem  Its own filesystem
                            Its own network     Its own network
                                    ↓                   ↓
                                Shared Linux Kernel
                                    ↓
                                Hardware

Both containers share the same kernel, but each thinks it has its own isolated environment. That is the magic - and why
containers are lightweight. There is no need to boot up a second OS, no second kernel to load. Just isolated processes
running on the same kernel.

Why VMs are different
--------------------------------------

A VM uses a hypervisor to simulate an entire computer - including a full hardware layer. Each VM runs its own complete
OS with its own kernel

                            VM A                    VM B
                            ────────────            ────────────
                            App A                   App B
                            Guest OS + Kernel A     Guest OS + Kernel B
                                    ↓                       ↓
                                Hypervisor (simulates hardware for each VM)
                                    ↓
                                Host OS + Kernel
                                    ↓
                                Hardware


This is the reason VMs are heavy - gigabytes of disk space, minutes to boot, significant memory overhead. You are
literally running two complete operating systems. Containers skip all this by sharing the host kernel.

The Windows/Mac complication
----------------------------------------

Since Docker is built on top of Linux container technology and Linux containers need Linux kernel features, how do i run
it on Windows or Mac?

DockerDesktop on Windows/Mac actually runs a lightweight Linux VM in the background - just enough to provide a Linux
kernel for the containers to use. You never see it, but it's there. That's why Docker Desktop uses more memory than
you might expect.

This is also why there are both Linux containers and Windows containers - they need different kernels.

How kernel links to code
----------------------------------------

When working with SecurityContext, it's the kernel that actually manages those threads. When Spring Boot creates a
thread pool to handle HTTP requests, it's asking the Linux kernel to create and schedule those threads. The kernel's
thread scheduler is what actually switches between threads on the CPU.

And when the JWT filter runs on a thread - the kernel is what gives that thread its isolated stack frame where local
variables live. The heap where your singleton beans live is managed by the JVM, which sits on the kernel.

                            Your JWT filter code (local variables on stack)
                                    ↓
                            JVM (manages heap, garbage collection)
                                    ↓
                            Linux Kernel (manages threads, memory, network)
                                    ↓
                            Hardware (CPU executes instructions, RAM stores data)



Containers Vs VMs
==========================================

            VMS                       |          Containers
-----------------------------------------------------------------------
         Your app                     |             Your app
         Guest OS(Full windows/linux) |             Container Runtime only
         Hypervisor                   |             Host OS Kernel(shared)
         Host hardware                |             Host hardware

A VM virtualizes the entire machine, including the OS - heavy, slow to start, gigabytes in size.

A container shares the host kernel and only packages what the application actually needs - lightweight, starts in
seconds, megabytes in size

Think of it like this:

* A VM is like renting an entire apartment - you get your own kitchen, bathroom, everything.
* A container is like renting a room in a shared house - you get your own space but share the infrastructure


The problem Docker solves
================================================

Without Docker, a new developer needs to:

* Install Java
* Install PostgreSQL 16 exactly
* Create the database
* Setup all the environment variables
* Hope OS doesn't cause issues
* Debug for hours when something doesn't match

With Docker:

* Install Docker
* Run a single command to build and run the application

That's it. The image contains everything. The environment is identical everywhere it runs - your laptop, new
colleagues machine or cloud server.

When using Docker, will you use 'localhost' for URLs, like for DB?
=======================================================================

When the Spring Boot application runs inside a container, 'localhsot' refers to those containers itself - not your machine,
and not the PostgreSQL container. Each container has its own isolated network identity. So 'localhost' from inside the
Spring Boot container points to nothing useful.

The way containers talk to each other is via service names defined in Docker Compose. For example:

services:
  app: # your Spring Boot app
  db: # your PostgreSQL

Docker creates an internal DNS system where each service is reachable by its name. So your Spring Boot container reaches
PostgreSQL not via 'localhost' but via: "jdbc:postgresql://db:5432/GymTracker" - no 'loalhost'

The service name 'db' resolves to the postgreSQL container's IP address automatically within the Dockers network.
No need to know actual IP, Docker handles that.

Crucial insight
---------------------------

Outside Docker  →  localhost:5432      (your machine's PostgreSQL)
Inside Docker   →  db:5432            (the PostgreSQL container)

This is why we use profiles. For Dev, we would use localhost, for local development. When running in Docker, you will
use different URL - which means new profile and env variable override.
