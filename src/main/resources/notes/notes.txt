

Threads (Real world example)
============================================

Think of server like restaurant kitchen. whe order comes in, a chef picks up order and handles it start to finish.

If ten orders come in simultaneously , ten chefs then handle the ten orders in parallel. Each chef has own workbench
- that's their stack, where local variables live. They don't share workbenches.

But, there is a shared fridge that all the chefs can use, this is the Heap, where objects like singletons beans live.
All chefs can reach into fridge at the same time, which is why mutable shared state ( weekNumber in ProgramWeekService )
is bad idea.

Now - for Security, we have something known as Security Context. Spring uses something know as ThreadLocal to store
theSecurityContext. A Thread Local is like giving each chef a personal safe pocket, that only they can access.
Chef A has User A identity and Chef B has User B identity. They are completely isolated,
even though they are in the same kitchen

Therefore, Spring reaches into each thread own personal safe pocket to get that requests authenticated user.
No sharing and no confusion between users


Understanding of use of Enums
==============================================

When storing data as a String such as "private String role = 'Admin'", there's nothing actually in place to stop this
 from happening:
 * user.setRole("Adminn") //typo
 * user.setRole("administrator") // different casing
 * user.setRole("GOD_MODE") // completely invalid

 Nothing prevent these errors happening at compile time. Bugs will only surface at runtime, which means potentially in
 production. Enum solves this by saying **these are the only valid values that exist for this concept, known
 at compile time**

 What an Enum actually is:
-------------------------------------------

 Enum is a special type in Java that represents a fixed set of named constants. Think of class, but with predefined
 and closed set of instances.

 Compiler now enforces validity:
 * user.setRole(Role.ADMIN) // valid
 * user.setRole(Role.ADMINN) // compile error - doesn't exist

 now you can never have invalid role

 Enums are more than just constants:
-----------------------------------------------

Enums are not just named integers but ar actually full objects. They can have fields. contractors and methods. Each contants
(USER, ADMIN) is actually instance of the enum class with its own data.

 Enums with JPA
-----------------------------------------------

When storing Enum in DB, there is two options:

1. @Enumerated(EnumType.ORDINAL) // stores 0, 1, 2... (the position)
2. @Enumerated(EnumType.STRING)  // stores "USER", "ADMIN"

Always use "STRING"

With Ordinal, if you ever reorder your enums or insert new on in the middle, every existing row in DB is now pointing
to incorrect role. Silent but catastrophic data corruption

Why split artifacts in pom.xml(jjwt-api,jjwt-impl,jjwt-jackson)?
=========================================================================

This is a deliberate design pattern called API/Implemetation separation.

Another example of the design pattern is: "List<Program> programs = new ArrayList<>();"

You code against the "List" interface but the actual work gets done by "ArrayList". If tomorrow Java introduces
better list implementation, you could swap ArrayList for it without changing ant of the code that uses List.

jjwt applies the same thinking:
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

* jjwt-api: the interface and the contracts you write your code against. Things like "Jwts.builder()", "Claims"...
This is what your code directly references and imports.

* jjwt-impl: the actual implementation of those interfaces(business logic of methods). the real logic that signs and
parses tokens. You never reference the classes directly in your code

* jjwt-jackson: handles JSON serialization/deserialization of the jwt payload using Jackson. JWT payloads are
JSON - something is needed to convert it

What does "<scope>runtime</scope>" mean in dependency?
=================================================================================

Maven controls when a dependency is available during the build lifecycle.

Most common scores includes - "compile(when application is building and starting up)",
"runtime(when application is running)", "test(fot tests)"

Scope       |       Available at compile        |       Available at runtime
--------------------------------------------------------------------------------------
compile     |                 Yes               |                   Yes
runtime     |                  No               |                   Yes
test        |                  No               |             Yes(in tests only)

So dependencies you directly reference in code(like jjwt-api mentioned earlier) will need to be available at compile
time but code that you don't directly use (jjwt-imp), there's no reason for it to be on the classpath - code doesn't need
it to compile successfully. Though at runtime, when code calls certain methods from let's say jjwt-api, JVM needs the
implementation of those methods to execute so it needs to be present at runtime

Marking dependencies as runtime scope enforces good practice.

Checked vs Unchecked Exceptions revision
=============================================================================

Checked Exceptions
----------------------------------

These refer to exceptions the compiler forces you to deal with. If a method can throw a checked exception, you must
either handle yourself(try catch) or declare you are passing the problem to caller(declaring a "throws {Exception}
in method").

The compiler refuses to compile code if you do neither. It's Java's way of saying "this operation can fail in a
meaningful way, you need to plan for it"

Unchecked Exception
------------------------------------

These extend RuntimeException. The compiler doesnt force you to handle or declare them.
They can propagate upwards in call stack silently until something catches them or the application crashes

Example:

// no throws declaration needed even though this could throw
public void doSomething() {
    throw new IllegalArgumentException("bad input"); // unchecked
}


Why are custom exception unchecked exceptions?
-----------------------------------------------

Exceptions in GlobalExceptionHandler consist of exceptions like ProgramNotFoundException and UserNotFoundException and
are ost likely extending Runtime exception

This is deliberate and Spring industry standard:

If they were checked, then every method in call chain would need to declare 'throws ProgramNotFoundException'

By making them unchecked, they propagate up automatically through Controller -> Service -> Repo and then eventually
handled by GlobalExceptionHandler - clean and no noise

How does Spring handles checked exceptions
------------------------------------------------------

Spring goes through process called Application Context Initialization, during this process it:
1. Scans for all beans marked my annotations
2. Creates instances of them in the right order based on dependencies
3. Wires dependencies together via injection
4. Calls @Bean methods to create and register those beans

So Spring calls all Beans during startup to create and register them

What happens if Bean throws exception
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

If a bean throws an exception during start up, spring catches it at th application context level and does this:

@Bean method throws Exception
           ↓
Spring catches it internally
           ↓
Wraps it in a BeanCreationException
           ↓
Application context fails to initialize
           ↓
Spring Boot shuts down with an error

Tells us a bean method threw exception and context couldn't be built. This will lead to application not starting up
as it will enforce us to resolve the issue as it sees it as a meaningful issue that needs to be dealt with.
So in the end when you use" throws Exception" in a method, you are technically saying - "I acknowledge this can fail,
If it does, I'm passing the problem to whoever called this - in this case the Spring application context - and
trusting it to handle startup failure appropriately"

What are Records and their uses?
=========================================================================

Java records are like classes except they have a lot less boilerplate and the fields are immutable, as they are
generated as private final fields which means once they have been assigned a value, they cannot be reassigned a new value.
For Records Java automatically generates:
1. Private final fields for each component
2. A constructor taking all components
3. getEmail() and getPassword() accessor methods
4.equals() and hashCode() based on all fields
5. toString()

It effectively eliminates all the boilerplate of a generic class.

Validation with Records
-----------------------------------------------------------

Validation annotations like @NotNull or @NotBlank work on records but syntax is slightly different as you annotate the
components directly.

public record LoginDto(
    @NotBlank @Email String email,
    @NotBlank String password
) {}

Though for any of those validation rules to apply, the @Valid annotation needs to be applied to the request body
parameter on you controller method.

Where should validation live, DTO or Entity?
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

Validation should live on the DTO, not Entity.

The Entity is a representation of the table in DB and should only reflect what's actually stored. Yuour DTO is your API
contract - it should enforce what's acceptable as input.

Also, if validation fails, better to catch it at the DTO before it reaches service or touches the DB. Data should
always be validated before it reaches the entity
